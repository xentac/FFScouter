# Filter box: track and unmount the previous React root on remount

`mountFilterBox` (`src/features/faction/index.ts`) calls `createRoot(container).render(...)` against a brand-new container on every call, with no reference kept to any previous root. Torn's faction/war page tabs are hash-routed (`#/tab=info`, `#/war/...`, confirmed via `should_run_faction()` and matching unit tests), so `on_navigation()` re-runs `process_page()` on every tab switch, and `process_page()` hands back a freshly-created `.members-list`/`.faction-war` element each time (the `data-ffscouter-initialized` re-entry guard lives on that element, so it never survives across navigations to prevent a second mount). The result: every tab switch calls `mountFilterBox` again, abandoning the previous mount with nothing ever calling `.unmount()` on it.

This is a real, unbounded, per-navigation memory leak, not just a lingering-but-harmless reference. `FFFactionFilterBox`'s mount effect (`src/ui/faction-filter-box.tsx:398-399`) registers `window.addEventListener("resize", ...)` and `window.addEventListener("ff-config-updated", ...)`; both are only removed by the effect's cleanup function, which only runs when React unmounts the component. Since `window` is permanent for the life of the tab, its listener list keeps a live reference into the closure chain: `onResize`/`onConfigUpdated` → `loadState` → `onFilterChangeRef.current` → the `onFilterChange` closure originally passed into `mountFilterBox` by `inject_filter_box`/`initialize_war_features`, which captures the `membersList`/`factionWar` element directly (e.g. `apply_filters_and_sort(membersList, ...)`). That pins the *entire* previous member/war list DOM subtree in memory — not just the small filter-box UI tree — once per tab switch, indefinitely. Over a long session with repeated tab switching, this is large enough to exhaust available memory (reported: WebExtensions process RSS growing from ~370MB to 7.86GB over 20-24 hours, crashing via OOM killer — [GreasyFork discussion](https://greasyfork.org/en/scripts/535292-ff-scouter-v2/discussions/335638)).

Fix: `mountFilterBox` tracks the current `Root` per mode (`"faction" | "war"`) in module-level state. Before creating a new root for a given mode, it calls `.unmount()` on the previous one for that mode, if any — severing `window`'s hold on the old closure chain so normal GC reclaims the abandoned component instance and everything it was pinning.

See [[Filter Box Remount Leak]] in `CONTEXT.md` for the glossary entry, and `TODO` for related candidates surfaced during this investigation that are not fixed by this change.

## Considered Options

- **Wrap the filter box's container in a custom element with `connectedCallback`/`disconnectedCallback`** (mirroring [[Settings Panel]]'s `FFSettingsPanel`, which already uses exactly this pattern correctly): more robust in principle, since the browser guarantees `disconnectedCallback` fires whenever the element leaves the document — including as part of a larger subtree Torn discards wholesale — without our code needing to detect that a swap happened at all. Rejected for now as a larger, more invasive change than the leak requires; left as a possible future direction if another remount-leak of this shape turns up elsewhere.
- **A document-level `MutationObserver` watching for `membersList`/`factionWar` removal, disconnecting the old root reactively**: rejected — this is strictly more machinery than a per-mode tracked reference, and the project already prefers the cheaper `Cleanup Interval` polling pattern over subtree observers for exactly this kind of teardown (see [[Cleanup Interval]]).
- **Do nothing / leave it to `cleanup_when_detached`**: not applicable — `cleanup_when_detached` tears down intervals/observers keyed off `el.isConnected`, but nothing in the codebase ever called it (or anything else) for the filter box's root.
