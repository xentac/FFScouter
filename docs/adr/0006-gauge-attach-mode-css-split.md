# Explicit attach mode for Gauge Marker vertical positioning

`.ffscouter-marker-wrapper` (the [[Gauge Marker]]'s positioning box) is attached to wildly different host elements depending on whether the player has honor bars enabled: a real bar graphic (`.honor-text-wrap`, ~14-19px tall) when enabled, or one of many unrelated name/row containers (`.member`, `.user.name`, `[class*="userInfoBlock__"]`, bazaar card variants, etc.) when disabled — none of which share a consistent height, even across rows matching the same selector.

Commit `994ddfa9` tried to serve both with a single CSS rule (`top: 50%; transform: translate(-50%, -80%)`), tuned to approximate the honor-bar look while "degrading gracefully" for taller containers. In practice this caused more overlap on the fallback side than it fixed, because the two cases don't just need different pixel values, they need different *techniques*: honor-bar wants a fixed cosmetic "hover above the bar" offset, while fallback wants pure vertical centering that doesn't depend on knowing the container's height in advance (an auto-margin trick — `top: 0; bottom: 0; margin: auto 0` — since heights aren't guaranteed stable even per selector).

We considered inferring the mode from the host's existing `.honor-text-wrap` class (already load-bearing everywhere else in the codebase for this same honor-bar/no-honor-bar branch) rather than adding a new parameter. We rejected it: the fallback side has no natural class of its own to key off, so a future third shape would have nowhere to go without revisiting every call site anyway. Instead, `GaugeAttachMode` (`HONOR_BAR` | `FALLBACK`) is passed explicitly by every call site and stamped as `data-ffscouter-attach-mode`, and the two CSS rules are kept fully separate rather than trying to unify them into one parameterized formula.
