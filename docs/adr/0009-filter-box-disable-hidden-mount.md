# Filter box disable settings: hidden mount, snapshot-level override, next-page-load

Two Settings-panel toggles, `ffconfig.faction_filter_enabled` and `ffconfig.war_filter_enabled` (both default `true`, preserving today's behavior), let users disable the [[Filter Box]] per mode. "Disabled" does **not** mean unmounted: `mountFilterBox` still runs and sets `display: none` on the container instead of `display: contents`. This is load-bearing — both FF/Est sort triggers (the box's own Sort button and native header clicks via `setup_header_click`) resolve a live `FactionFilterBoxHandle` through `getFilterBoxHandle`, and with no mounted box `getFilterBoxHandle` returns `null`, silently killing FF/Est sorting with no other code path to restore it. Losing sort along with the filter was ruled unacceptable; header-click sort (headers already show a pointer cursor) remains the sort affordance while the box is hidden. Column display stays configurable via the Settings panel's own "Faction/War Page Shows:" selects, which `column-population.ts` reads independently of the box.

Filtering is neutralized by reusing the engine's existing `filterEnabled` short-circuit (`filter-sort-engine.ts` shows every row when the snapshot's `filterEnabled === false`): a `filteringDisabled` prop, seeded from the mode's config in `mountFilterBox`, makes the component's single snapshot builder (`getFilterSnapshot`) force `filterEnabled: false`. Forcing at snapshot *production* matters because snapshots are consumed on two paths — the `onFilterChange` callback and direct `handle.getFilterSnapshot()` calls from the mutation-observer reapply watcher — and forcing only at the callback would leave the observer path applying stored filters invisibly. The stored `faction_filter_state` / `war_filter_state` blobs are never touched: only the *reported* snapshot lies about `filterEnabled`, so re-enabling the filter in Settings restores the user's exact prior filter configuration. Inside the component the prop is read through `filteringDisabledRef`, matching the existing ref convention (`modeRef`, `filterStateRef`) for values read from imperative-handle and debounce closures.

The toggle takes effect on the next page load, not live. The Settings panel lives on `profiles.php` while the filter box lives on faction pages, so the same-page `ff-config-updated` event can never reach a live box in normal use, and cross-tab messaging is not practical in a userscript. SPA navigation back to a faction page re-mounts the box and picks up the new config.

## Considered Options

- **Not mounting the box at all when disabled**: rejected — silently removes FF/Est sorting (both sort entry points require a mounted handle), a regression beyond what "disable the filter" promises.
- **Rendering a slimmed box (Sort button + display select only)**: rejected — the intent of the setting is that no FFScouter filter UI appears on the page.
- **Flipping the component's actual `filterEnabled` state on mount (with persistence suppressed)**: rejected — the state writes through to the persisted filter-state blob, so suppressing that write means special-casing the component's state-write path; the snapshot-level override reuses the same downstream machinery without touching storage.
- **Live update via `ff-config-updated`**: rejected — the event is same-page only and the two UIs are on different pages; building cross-tab messaging for this isn't worth it.
