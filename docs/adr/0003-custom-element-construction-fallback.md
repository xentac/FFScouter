# Custom element construction: check instanceof and retry, then degrade gracefully

With Adblocker Ultimate enabled in Firefox (Tampermonkey, native `browser.userScripts` API injection, any injection mode), `document.createElement()` for our Lit-based custom elements (`ff-settings-panel`, `ff-header-line`, `ff-flight-profile-status`, `ff-faction-filter-box`) intermittently returns a broken, non-functional element instead of a real instance — visible on screen as an empty box, with `TypeError: this._$Ev is not a function` (Lit's `ReactiveElement` constructor bootstrap) reported to the console. Per the HTML spec, a throwing custom element constructor does not make `createElement` throw — the "create an element" algorithm reports the exception and substitutes a generic fallback element instead, so a `try`/`catch` around `createElement` never has anything to catch. `create_ff_element()` (`src/utils/dom.ts`) instead checks whether the result is actually `instanceof` the registered class, retrying a few times with short delays before giving up and returning `null`; callers (`settings`, `profile`, `attack`, `profile-flights`, `mini-profile-flights`, `faction`) bail out cleanly and the settings panel surfaces a toast when this happens.

We deliberately stopped at graceful degradation rather than chasing a full fix. The actual interference happens inside Tampermonkey's native-userScripts-API bridge (`page.js`/`content.js`, Firefox's `ext-userScripts-content.js`), a layer FFScouter's own code cannot reach or instrument — retries never succeed while Adblocker Ultimate is active, confirming the failure is a persistent condition for the page's lifetime, not a transient race. Ruled out along the way: double script injection (real bug, fixed separately, but not the cause here), duplicate Lit/`@lit/reactive-element` copies in the bundle, iframes, a second top-level navigation, and a patched `customElements.define`/`HTMLElement`.

## Considered Options

- **Catch the exception and retry**: rejected — per the HTML spec, the constructor's exception never propagates to the `createElement` caller, so there's nothing to catch.
- **Chase the Tampermonkey/Adblocker Ultimate interaction further** (global "disable native userScripts API" toggle, bug reports upstream): rejected for now — outside what this codebase can control or verify; revisit if it recurs with other extensions.
- **Site-level workaround** (allowlist torn.com in Adblocker Ultimate): viable for affected users today, but isn't a code fix and isn't something FFScouter can enforce.
