# Settings panel own-profile scoping: opt-in, checked in `run()`, asymmetric fail-open/fail-closed

The [[Settings Panel]] mounts unconditionally on every `profiles.php` load today (`shouldRun()` is just `torn_page("profiles")`), regardless of whose profile is being viewed. We added an opt-in toggle, `ffconfig.settingsPanelOwnProfileOnly` (default `false`, preserving today's behavior), that restricts it to the logged-in user's own profile: `extract_id_from_url(window.location.href)` (the viewed profile's id) is compared against `parseInt(await getLocalUserId(), 10)` (the logged-in user's own id, read from the native Torn `.settings-menu` "View Profile" link) — the exact primitive `status-attack` already relies on for self-attack prevention, so it isn't a new trust assumption.

The two ways this comparison can come up empty are handled asymmetrically, on purpose. If `getLocalUserId()` can't resolve the logged-in user's own id (the DOM lookup fails or times out — rare, since the same lookup already backs a correctness-sensitive check elsewhere), we fail *open* and show the panel: this is an opt-in convenience restriction, and silently vanishing the entire settings UI with no visible cause is a worse outcome than the restriction occasionally not applying. If instead `extract_id_from_url` finds no `XID` at all on the *current* page, we fail *closed* and hide the panel — confirmed this means Torn's own "profile not found" error state, not an implicit self-view, so there is no profile here to call "the user's own."

The check lives at the top of `run()`, before `create_ff_element` and all event wiring — not in `shouldRun()`. `index.ts`'s boot loop `await`s every feature's `shouldRun()` sequentially, and `getLocalUserId()` carries a 15s `wait_for_element` timeout on its failure path, so putting the check in `shouldRun()` would let one slow/absent DOM lookup stall every feature queued after Settings. This mirrors `status-attack`, which already only ever calls its equivalent (`fetchLocalUserId()`) from `run()`, never `shouldRun()`, for the same reason.

## Considered Options

- **Converting the boot loop to run all features' `shouldRun()` concurrently** (which would also make gating in `shouldRun()` safe): rejected as out of scope here. No ADR or commit history explains why the loop is sequential, and we found no invariant that actually depends on it today (HTTP interceptor order, the one plausible candidate, is independently guaranteed by priority sorting in `network.ts`, not registration order) — but it's a change to shared boot behavior affecting every feature, not a one-file fix, and deserves its own decision if pursued later.
- **Gating directly in `shouldRun()`**: rejected — blocks the sequential boot loop for every feature queued after Settings on a slow or absent DOM lookup.
- **Failing closed on every detection failure** (both `getLocalUserId()` and URL-id misses): rejected for the `getLocalUserId()` case — an opt-in restriction silently eating the whole panel with no visible cause is worse than rarely failing to apply it.
