# React fallback: @require Torn's own react-dom build, not fetch+eval it ourselves

[[Page React]] (`src/shims/react.ts`, `src/shims/react-dom.ts`) borrows `unsafeWindow.React`/`.ReactDOM` instead of bundling `react`/`react-dom` into the `.user.js`, saving ~500KB+ of shipped code. On at least one Safari/Tampermonkey configuration this is permanently broken (every load, not a timing race): `unsafeWindow` resolves to a genuinely separate realm (`unsafeWindow === window` is `false` there ŌĆö confirmed via a standalone diagnostic userscript run on the reporting user's machine), and `.React`/`.ReactDOM` are `undefined` on it ŌĆö that userscript manager's isolated sandbox doesn't bridge arbitrary page-realm JS *objects*, even though plain DOM access (`document.scripts` etc.) stays reliable.

We first tried fetching Torn's own `react-dom` UMD build (`torn.com/builds/react-umd/react-dom.<version>.<hash>.production.js` ŌĆö React itself is bundled inside it as an internal webpack module) via `GM_xmlhttpRequest` and running it with `new Function(...)` in our own realm, discovering the exact URL at runtime by scanning `document.scripts`. The same diagnostic userscript that confirmed the realm split above also proved this doesn't work, in two independent ways on two different engines:

- **Chrome**: `new Function(...)` throws `EvalError: ... 'unsafe-eval' is not an allowed source of script`. Torn's CSP `script-src` includes `'unsafe-eval'` on some page loads but not others (the directive is dominated by ad/tracking-network domains, so it's plausibly tied to which third-party ad SDK is active on that particular route) ŌĆö confirmed by capturing two `content-security-policy` headers from the same site, one with and one without `'unsafe-eval'`. There's no nonce/hash workaround: CSP nonces and hashes only ever authorize `<script>` *elements*, never bare `eval()`/`new Function()` calls, which are gated solely by the `'unsafe-eval'` keyword.
- **Safari**: the diagnostic's fetch-and-eval test reported `evalOk: true` (no CSP block) but `window.React`/`.ReactDOM` were still `undefined` immediately afterward, read from the same script that ran the eval. The dynamically-evaluated code's `self`/`globalThis`/`window` resolution landed somewhere other than the realm our own script observes as `window` ŌĆö an engine-specific quirk in how Safari's userscript sandbox implements `new Function`, not something we can rely on.

Two different engines, two different failure modes, same underlying problem: `eval`/`new Function` execution semantics aren't well-defined enough across userscript-manager sandboxes to build on.

**Decision**: use `@require` instead. It's a standard userscript metadata directive (`vite.config.ts`'s `userscript.require`) that has one well-defined guarantee across Tampermonkey/Violentmonkey: the required script's source is run in the *same realm* as the main script, before any of the main script's own code ŌĆö no ambiguity about which `window` receives the result, no page-CSP dependency (the userscript manager fetches and injects it directly, the same privileged path that delivers the main script itself, not a page-context network request or `eval`). We point it at the exact same Torn `react-dom` UMD URL rather than building or hosting our own copy, so the fallback always matches whatever React version Torn's own page runs. That URL has no version-controlled path we own ŌĆö the hash is Torn's own build artifact name and changes on their deploys ŌĆö so it needs manual bumping in `vite.config.ts` when the fallback stops working (check a Torn page's `<script src>` tags matching `/builds/react-umd/react-dom.*.production.js` for the current one).

Because `@require` runs before any of our own module code, `getReact()`/`getReactDOM()` (`src/shims/react-loader.ts`) stay fully synchronous: prefer `unsafeWindow.React`/`.ReactDOM` when both are present, otherwise read the same-realm globals the `@require`d bundle already set by the time our code runs. No async gate, no memoized fetch state, no mount call site needs to change.

Torn PDA doesn't support `@require` at all ŌĆö its script loader just ignores the directive, which is fine, since PDA already runs unsandboxed (`unsafeWindow` is literally `window` there) and has never hit this bug.

`ffconfig.debug_force_react_fallback` (Settings > Debug Settings) forces every mount through the `@require`d fallback even when `unsafeWindow` already works, so it can be exercised on Chrome/Firefox ŌĆö where it's normally never triggered ŌĆö ahead of a Safari-specific release.

## Considered Options

- **Bundle `react`/`react-dom` into the userscript directly**: rejected ŌĆö `react-dom/client`'s production build alone is ~536KB minified; would roughly double/triple the shipped bundle for every user to fix a problem only some Safari users hit.
- **Force `sandbox: 'raw'` (whole script executes in the page's real JS context)**: rejected ŌĆö collapses the isolation boundary for `GM_xmlhttpRequest`, config storage, and everything else, not just React. Torn PDA already runs this way (no sandbox at all) and demonstrates the hazard class directly: `customElements.define("ff-settings-panel", ...)` at module top level threw `NotSupportedError` on reinjection, because there's no fresh isolated realm to hide a rerun behind. Fixed independently (guarded with `customElements.get()`), but the bug surfaced specifically while evaluating this option.
- **Dual-context bridge** (inject only the React-consuming render layer into the page via a `<script>` tag, talk to the sandboxed side over `CustomEvent`s): rejected ŌĆö technically viable (Torn's CSP requires a nonce for inline scripts, but the nonce is readable off any of Torn's own already-rendered `<script nonce>` tags) but requires rebuilding every React-consuming component to route props and callbacks through a serializable event protocol, since functions can't reliably cross the isolation boundary, plus the same reentrancy discipline the `customElements` bug showed we don't consistently have yet. Disproportionate for the problem.
- **Fetch-and-`eval` Torn's UMD build ourselves** (the first attempt): rejected after direct testing ŌĆö see above. Not fixable without either giving up on same-realm execution (defeating the point) or depending on a CSP grant that's confirmed to vary per load.
- **Build our own React 19 UMD bundle and host it (CDN or self-hosted)**: viable in principle ŌĆö React 18+ dropped official UMD builds from npm, so this would've meant building and hosting a custom one, which is exactly what `@require`ing Torn's own equivalent file gets for free, with guaranteed version parity to the page and no separate hosting/build step to maintain.
- **React 17's official unpkg/jsdelivr UMD build**: rejected ŌĆö no `createRoot` (introduced in React 18), and this codebase's mount call sites all use `createRoot`.

## Consequences

- The `@require`d bundle runs on every page load for every user, not just when the fallback is needed (there's no way to make `@require` conditional) ŌĆö a small extra parse/execute cost for everyone, accepted as worthwhile.
- The exact `@require` URL is a manual-maintenance liability: it will silently go stale whenever Torn redeploys their react-dom build, until someone notices and updates `vite.config.ts`.
