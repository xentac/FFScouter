# Plain functions over plain data for estimate resolution, not a class

get-stats now returns [[Available Estimates]] — three per-[[Estimate Source]] candidates cached alongside the merged top-level fields, so a future priority policy (age cutoff, premium-over-standard) can re-rank already-cached data without a refetch. Picking a winner among them needed a single place to live, and a class wrapping `FFDataComplete` with `.ff()`/`.bsEstimate()`-style methods was considered as that place, mainly to memoize the pick across the several `extract_*` calls a single UI element makes on the same record.

Decision: [[Estimate Resolution]] (`resolve_estimate` in `src/utils/estimate.ts`) and the `extract_ff`/`extract_bs_estimate`/`extract_bs_estimate_human`/`extract_source`/`extract_last_updated` accessors built on it are plain functions over the plain `FFDataComplete` interface — no class, no memoization. Two reasons: `ffcache.ts` stores `CachedFFData` in IndexedDB via `idb`, which persists values through the structured clone algorithm; structured clone does not preserve class prototypes, so a class instance written to the cache comes back out as a plain object missing every method — the wrapper would need to be reconstructed on every cache read anyway, and would silently stop working wherever that reconstruction was missed. Separately, the resolution work itself is a handful of field comparisons across at most three candidates — running it 3-4x per row (once per `extract_*` call) is cheaper than a single DOM write, so the memoization a class would buy costs more in allocation (both the wrapper instances and the IndexedDB rehydration) than it saves. This also matches every other data shape already in the codebase (`FFData`, `Flight`, `PlayerFlightsResponse`) — `FFScouter` and `FFApiError` are the only classes, and both are stateful orchestrators/exceptions, not per-record data wrappers.
